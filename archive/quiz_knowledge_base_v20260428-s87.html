Coverage review policy — required after every stats upload After every session that includes a stats JSON upload, Claude must produce a comprehensive coverage review before writing any new questions. The review must include: Category-level breakdown: question count, level distribution, type distribution (gap/input/mcq/multi), input ratio vs 20% target Phrasal verb family audit: which verb families are covered, which forms are missing within each family Grammar structure gap check: cross-reference against B1–C1 curriculum checklist Teacher session alignment: confirm which specific lesson items (errors, phrases, structures) are in the bank vs still missing Prioritised recommendation table: ranked by impact, with estimated question counts per gap Only after Artem has confirmed the priorities should Claude proceed to drafting questions. New questions are always shown as a draft table for review before building the deploy. This workflow applies even if Artem asks to "just add more questions" — always run the review first when fresh stats are available. Deployment Workflow At the end of each Claude session, Claude generates a deploy.html with the latest index.html and sw.js embedded as base64. Artem downloads it, opens it in Chrome, clicks Deploy. That's it. Step 1 Claude generates deploy.html → Step 2 Download & open in Chrome → Step 3 Click Deploy → Step 4 GitHub Pages live in ~60s Important deploy.html is regenerated every session — it is not stored anywhere between sessions. Always generate a fresh one before asking Artem to deploy. Post-s82: deploy.html is ~1.3 MB (questions inline). This is the same size as pre-s78 and works fine — SW caches it after first load. Why Git Trees API (not PUT /contents) Two sequential PUT /repos/.../contents/file calls create two commits → GitHub Pages triggers two builds → first is cancelled → Artem receives a "workflow cancelled" email. The Git Trees API creates a single atomic commit with both files → one build → no emails. Version string Format: v{YYYYMMDD}-s{N} where N is the session number matching the KB changelog entry. Same-session rebuilds append r2, r3 etc. Embedded in the SW cache name so each deploy busts the old cache automatically. Must be identical across: index.html badge · SW cache key · deploy filename · KB filename. Deploy script verification — mandatory (added S65, post-bug) After generating deploy.html and before presenting to Artem, verify: var I='...' contains >100 base64 chars (catches empty/truncated data from Python string bugs) var S='...' contains >100 base64 chars SW cache key matches deploy version: decode var S, confirm CACHE='eq-{version}' matches the deploy version string. Never reuse SW base64 from a prior deploy — always rebuild SW with the current version. Stale SW cache key = browser serves old JS regardless of new HTML. (Root cause s66r1–r4 blank card bug.)
